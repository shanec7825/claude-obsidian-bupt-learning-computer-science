# 贪吃蛇图形化版 · 演示讲解文档

> 面向课堂答辩 / 项目展示。建议演示时长：**8-12 分钟**。
> 配合实际运行程序进行讲解，本文档提供每一页的口述要点。

---

## 第一页：项目概述（1 分钟）

### 我们做了什么

一个基于 **raylib** 图形库的贪吃蛇游戏，支持三种模式。

| 模式 | 操作 | 一句话描述 |
|------|------|-----------|
| **经典模式** | WASD | 一个人玩，吃食物变长，撞墙撞自己就死 |
| **双人对战 (PvP)** | P1=WASD, P2=方向键 | 两个人同键盘对战，碰头即死，杀对方变长 |
| **人机对战 (Battle)** | WASD | 和 AI 打，AI 会 BFS 寻路，死了还会重生 |

**口述要点：**

> "我们做的是一个图形化贪吃蛇。不是控制台里用 `printf` 打印的那种——有窗口、有彩色棋盘、蛇头带眼睛、食物会脉冲发光。"
> 
> "三种模式：经典的一个人玩、PvP 两个人同一键盘对战、Battle 人机对战。AI 有三种难度可选。"

### 演示动作

- **启动程序** → 展示主菜单界面
- 鼠标不碰，纯键盘操作，强调"全键盘控制"

## 第二页：系统架构（1.5 分钟）

### 我们怎么组织的代码

**口述要点：**

> "项目总共 13 个文件，分成 7 个模块。核心思路是'分层架构、各司其职'。"
> 
> "最顶层 `main.cpp` 只做三件事：开窗口、场景调度、关窗口。它不知道游戏怎么玩——它只知道'现在是菜单还是游戏中'。"

```
┌─────────────────────────────────────────┐
│              main.cpp                    │
│     "现在是什么场景？要不要切换？"         │
├─────────┬─────────┬─────────┬───────────┤
│ scene   │  game   │ render  │  audio    │
│"场景"   │"逻辑"   │"渲染"   │"音效"     │
├─────────┴─────────┴─────────┴───────────┤
│         ai (AI寻路)    input (键盘)       │
├─────────────────────────────────────────┤
│              raylib 库                   │
└─────────────────────────────────────────┘
```

**关键概念——抽象基类：**

> "场景切换用到了 C++ 的**多态**。我们定义了一个抽象基类 `Scene`，它只规定'每个场景必须能 update、render、返回下一步去哪'，但不写实现。"
> 
> "`MenuScene` 和 `PlayScene` 各自实现这些函数——但 `main.cpp` 不需要知道当前是哪个场景，它只需要调 `current->update()`。加第四个模式不需要改 main，这就是面向对象的好处。"

## 第三页：数据结构——GameState（1.5 分钟）

### 整个游戏的状态存在一个结构体里

**口述要点：**

> "有一个核心结构体叫 `GameState`，约 30 个字段，整个游戏的所有数据都在里面。"

```cpp
typedef struct gamestate {
    char map[40][40];          // 35×35 字符棋盘
    // P1 蛇
    Point snake[1600]; int length, score;
    // P2/AI 蛇（字段复用）
    Point ai_snake[1600]; int ai_length;
    // 控制
    bool started, paused, game_over;
    int pending_growth;        // 击杀奖励
    bool p1_drop_requested;    // 尾部丢弃
} GameState;
```

**为什么两条蛇放一个 struct 里？**

> "有人会问——为什么不定义 `Snake` 结构体然后声明两个？答案是碰撞检测。两条蛇的位置最终都在 `map[][]` 上——P1 头要去哪，直接读 `map[row][col]` 就知道那里是空地、墙还是对方身体。O(1)。如果拆成两个独立对象，每次碰撞检测都要遍历对方蛇身。"

**演示动作：**

- 打开 `game.h`，指出 `GameState` 结构体
- 重点指 `pending_growth` 和 `p1_drop_requested` 两个字段——"这两个是我们自己设计的创新机制"

## 第四页：地图字符编码（1 分钟）

### 棋盘上每个字符都有含义

| 字符 | 含义 | 颜色 |
|------|------|------|
| `#` | 墙 | 深灰 |
| `.` | 空地 | 棋盘格交替 |
| `F` | 食物 | 红色脉冲 |
| `H` `B` | P1 蛇头/身 | 绿色渐变 |
| `A` `C` | P2/AI 蛇头/身 | 蓝色渐变 |
| `O` | 障碍物 | 中灰 |
| `D` `E` | P1/P2 丢弃方块 | 绿/蓝 + 白叉 |

**口述要点：**

> "注意 `D` 和 `E`——这是我们设计的**尾部丢弃**机制。按 G 键，P1 蛇尾掉一节，变成地图上的 `D` 方块。`
> "有趣的是：`D` 对 P1 来说是'食物'——可以吃回去。但对 P2 来说 `D` 是障碍物，撞上就死。"

## 第五页：核心算法——碰撞检测（2 分钟）

### 11 步优先级，顺序不能乱

**口述要点：**

> "这是整个项目最复杂的一段逻辑。双人对战中，两条蛇同时移动，碰撞检测有 11 个步骤，**顺序严格不能改**。"

**演示动作——打开代码，指 `move_pvp_snakes`：**

> "第一步——头碰头。两个蛇头冲向同一格，同归于尽。如果这一步不放在最前面，而先判 P1 撞 P2 身体，P1 死但 P2 活——不公平。"
> 
> "第二步和第三步——攻击对方身体。为什么在自撞前面？因为攻击对方是'对战行为'，应该优先于'自己不小心'。"
> 
> "第六步和第七步——自撞。为什么在最后？因为前面如果已经判了 P1 死（撞到 P2 身体），`!p1_dies` 条件会阻止重复判断。"

**口述要点——尾部安全：**

> "还有一个精巧的例外——尾部安全。如果蛇这一帧不吃食物也不自动增长，尾巴会前移一格。所以蛇头可以安全地走到'即将被清空的尾巴位置'。判断条件是 `!will_grow && next == body[length-1]`。"

## 第六页：创新机制（2 分钟）

### 机制一：击杀奖励（pending_growth）

**口述要点：**

> "PvP 或 Battle 中，你杀了对方蛇，不会瞬间变长——那样画面会'爆炸'，而且大概率下一帧就自撞死了。"
> 
> "我们用 `pending_growth` 记录奖励点数。假设对方长 120 节，你的 `pending_growth` 变成 120。之后每一步消耗 1 点——蛇每走一步长一节。吃食物和自动成长的回合不消耗。"
> 
> "效果：击杀后的一两分钟里，你的蛇不断变长，地图越来越挤。'大蛇'的压迫感就是这样来的。"

### 机制二：尾部丢弃（Drop-Tail）

**口述要点：**

> "按 G 键，丢弃尾部一节，变成障碍方块。条件是蛇长大于 3 节。"
> 
> "有趣的是——如果你同时吃到了食物，增长和丢弃互相抵消。蛇不长也不丢，但分数照加。这是我们故意设计的：用一次 discard 换一次'零增长得分'。"

**演示动作：**

- Battle 模式让蛇长到 10+ 节，按 G 展示丢弃
- 指出地图上出现的 `D` 方块和白色 "×" 标记

### 机制三：独立双人定时器

**口述要点：**

> "PvP 中两个玩家有各自独立的移动定时器。正常都是 160ms 一步。但按住 F 可以 4 倍加速——只有 P1 加速，P2 不受影响。"
> 
> "这意味着 P1 可以比 P2 快 4 倍。这在经典的'共用定时器'设计里是做不到的。"

## 第七页：AI 设计（1.5 分钟）

### 三级难度，BFS 寻路

**口述要点：**

> "AI 不是'随机乱走'。核心是 BFS 广度优先搜索。"

| 难度 | 寻路方式 | 特殊行为 |
|------|---------|---------|
| Easy | 曼哈顿距离贪心 | 30% 概率随机走（显得"笨"） |
| Normal | BFS 最短路径 | 15% 概率随机走 |
| Hard | BFS + 安全检测 | 蛇长 ≥105 时检查"走了这步还能不能绕回来" |

**口述要点——Hard 安全检测：**

> "Hard 难度的核心创新。当蛇很长时（≥105 节），AI 会模拟——'如果我往前走这一步，还能不能 BFS 到达我的尾巴？'如果不行，说明走入死胡同，这步放弃。"

**演示动作：**

- 切到 Battle Hard 模式
- 观察 AI 在蛇长超过 100 后行为变化——不再追食物，优先保命

## 第八页：程序化音效（1 分钟）

### 没有音频文件，全部代码合成

**口述要点：**

> "四个音效——吃食物、死亡、开局、移动——不需要任何 .wav 或 .mp3 文件。全部用正弦波公式在代码里算出来。"

| 音效 | 公式 | 感觉 |
|------|------|------|
| 吃食物 | 440Hz → 880Hz，150ms | 短促上升，"叮" |
| 死亡 | 400Hz → 120Hz，500ms | 下降，"嘟——" |
| 开局 | C5→E5→G5 琶音，300ms | 三连音上行 |
| 移动 | 120Hz 固定，30ms | 极其短促，"嗒" |

**口述要点：**

> "所有音效带线性衰减包络——音量从 100% 平滑降到 0%，防止播放结束时'啪'的一声爆音。"
> 
> "PCM 参数：22050Hz 采样率、16-bit、单声道。内存占用极小。"

## 第九页：项目总结（1 分钟）

### 技术栈一览

| 方面 | 技术 |
|------|------|
| 语言 | C++（C 风格结构体 + C++ 类多态混合） |
| 图形库 | raylib 5.x |
| 窗口 | `InitWindow` / `SetWindowSize` / `CloseWindow` |
| 渲染 | `DrawRectangleRounded` / `DrawCircle` / `DrawTextEx` |
| 输入 | `IsKeyPressed`（边沿）/ `IsKeyDown`（电平） |
| 音频 | PCM 正弦波合成 + `LoadSoundFromWave` |
| 字体 | `simhei.ttf` → `msyh.ttc` → 默认，按需码点加载 |
| 持久化 | `fopen` / `fscanf` / `fprintf`（最高分） |
| 平台 | Windows（通过 raylib 可移植到 Linux/macOS） |

### 规模

| 指标 | 数值 |
|------|------|
| 源文件 | 13 个（7 .cpp + 6 .h） |
| 代码行数 | ~3100 行 |
| 函数数量 | ~35 个 |
| 模块数 | 7 个 |
| 游戏模式 | 3 种 |
| 地图字符类型 | 10 种 |

### 我们学到了什么

> "这个项目让我们真正理解了："
> 1. **多态不是课本概念**——Scene 抽象基类 + MenuScene/PlayScene 就是多态的真实应用。
> 2. **数据决定算法**——GameState 的设计直接影响碰撞检测的效率。"两蛇同 struct"不是因为懒，是因为 `map[][]` 作为共享现实让碰撞检测 O(1)。
> 3. **边界条件最难**——11 步碰撞检测的优先级顺序、pending_growth 和丢弃同时发生的逻辑，这些边界 case 花了最多调试时间。
> 4. **没什么是'写完就对的'**——从 EasyX 迁移到 raylib、从单一模式扩展到三种模式，每一步都需要重新审视之前的假设。

### 如果被问到的问题（预案）

**Q: 为什么用 raylib 而不是更流行的 Unity/Unreal？**
> raylib 是一个纯 C 的轻量图形库，适合学习底层原理。Unity 屏蔽了渲染管线——你拖一个 Sprite 不用关心它是怎么画到屏幕上的。raylib 让我们自己管理每一帧的 `BeginDrawing` / `EndDrawing`，对理解游戏循环更有帮助。

**Q: 为什么不把蛇抽象成 Snake 类？**
> 当前项目用的是 C 风格的结构体 + 全局函数。用 Snake 类封装蛇身、方向、增长逻辑会更"面向对象"。我们选择不这样做是因为 GameState 作为统一的"棋盘状态"快照，让碰撞检测只需要查 `map[][]` 就能 O(1) 知道任意格属于谁。拆成独立对象后这个优势就没了。

**Q: 如果再加一种模式，要改哪些文件？**
> 1. `game.h`：加 `MODE_NEW` 枚举值，可能加新字段到 GameState
> 2. `game.cpp`：写 `init_new_game()` 和 `move_new_snakes()`
> 3. `scene.cpp`：`MenuScene::update` 加一个按键，`PlayScene::update` 加一个 `else if`
> 4. `render.cpp`：`draw_panel` 加新模式的信息面板布局
> 
> main.cpp **不用改**。这就是多态架构的价值。

**Q: AI 能再强一点吗？**
> 当前 Hard 的 BFS 是"最短路径寻食物 + 尾部安全检测"。更强的 AI 可以做"前瞻搜索"——不只模拟一步，而是模拟 3-5 步后的局面。或者用 A*（启发式搜索）替代 BFS，用"到食物的距离 + 到尾巴的距离"做启发函数。但计算量会显著增加——每帧要评估几十个候选路径。
