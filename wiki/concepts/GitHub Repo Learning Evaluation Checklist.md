## 仓库三层解构

任何 GitHub 仓库都可以按功能拆成三层。学习时**只深入核心代码层**，环境层和教学设计层只需"知道存在+能跑通"。

```
📦 仓库根目录
│
├── 🔧 环境层（跑通即可，不深入读）
│   ├── CMakeLists.txt / Makefile    → 构建脚本
│   ├── lib/ / vendor/ / deps/       → 第三方依赖
│   ├── .vscode/ / .github/          → 编辑器/CI 配置
│   ├── .gitignore / .gitattributes  → 版本控制配置
│   └── LICENSE / CODE_OF_CONDUCT    → 项目元数据
│
├── 📚 教学设计层（理解意图用，不是学习材料本身）
│   ├── README.md                    → 项目概述、构建说明
│   ├── 视频/博客教程链接            → 脚手架
│   ├── 注释中的设计意图             → "为什么这样做"
│   └── CONTRIBUTING.md              → 贡献指南
│
└── 🧠 核心代码层（你要深入读的）
    └── src/ 或 项目名/
        ├── main.cpp / main.c        → 入口点
        ├── 核心数据结构             → 类、结构体
        ├── 算法/业务逻辑            → 函数实现
        └── 测试（如果有）           → 验证理解的工具
```

### 学习时的操作顺序

1. **跑通环境层**（5 分钟）：Clone → 按 README 构建 → 看到程序运行
2. **扫一眼教学层**（10 分钟）：理解项目做什么、有什么模块
3. **只读核心代码层**（主要时间）：用结构化阅读六步法逐个文件读
4. **遇到环境层的语法不纠结**：`CMakeLists.txt` 里不懂的语法跳过——那是工具知识，不是编程知识

## 新手友好度评估清单

对一个仓库打分（每项 1-3 分，总分 / 项数 = 推荐指数）：

### 必须条件（不满足 = 不推荐）

| 条件 | 检查方式 |
|------|---------|
| □ 能本地跑通 | Clone → 按 README 构建 → 看到程序运行或测试通过 |
| □ README 有构建说明 | 打开 README，搜索 "build" / "install" / "run" |
| □ 最近 12 个月有提交 | GitHub 首页看 "last commit" |
| □ 有 License 文件 | 根目录找 `LICENSE` 或 `LICENSE.md` |

### 代码质量评分（每项 1-3 分）

| 评分项 | 1 分 | 2 分 | 3 分 |
|--------|------|------|------|
| **文件数量** | 20+ 文件，不知道从哪开始 | 5-10 文件，有清晰入口 | 3-5 文件，单一职责 |
| **代码注释** | 无注释或全中文拼音 | 有 docstring，但不多 | 解释"为什么"而非"是什么" |
| **命名质量** | `a`, `b`, `tmp`, `x1` | 能猜出来意但不精确 | `calculate_total_price` |
| **函数长度** | 200+ 行单函数 | 50-100 行，逻辑混合 | <50 行，单一职责 |
| **测试** | 无测试 | 有测试文件但覆盖率低 | 测试覆盖核心逻辑 |
| **构建简单度** | 需要配置一堆东西 | 标准 CMake/Makefile 但依赖多 | 一行命令构建 |

### 学习匹配度评分（每项 1-3 分）

| 评分项 | 1 分 | 2 分 | 3 分 |
|--------|------|------|------|
| **技术栈匹配** | 完全陌生语言/框架 | 语言熟悉，框架陌生 | 语言+框架都熟悉 |
| **难度梯度** | 跳崖式——完全看不懂 | 能懂 30-50%，需要大量查资料 | 能懂 60-70%，在能力边缘 |
| **视觉反馈** | 纯库/API，无可见输出 | 命令行输出 | 图形/游戏——结果立即可见 |
| **教学支持** | 无 README，无教程 | README 基本清楚 | 有视频/博客教程逐行讲解 |

### 综合推荐指数

```
总分 / 10 项 × 10 = 推荐指数（满分 100）

≥ 80: 强烈推荐，立刻开始
60-79: 可做备选
< 60: 换一个
```

## 识别"环境噪音"——哪些是你不需要读懂的

当你在源码目录看到以下文件/目录时，它们属于环境层或工具层，**初学时不需要深入理解**：

| 文件/目录 | 它是什么 | 你需要做什么 | 不需要做什么 |
|-----------|---------|-------------|-------------|
| `CMakeLists.txt` | CMake 构建配置 | 知道 `cmake . && make` 就行 | 不需要读懂 CMake 语法 |
| `Makefile` | Make 构建规则 | 知道 `make` 或 `make run` | 不需要学 Make 语法 |
| `lib/` 或 `vendor/` | 第三方库 | 知道项目依赖了哪些库 | 不需要读库的源码 |
| `.vscode/` | VS Code 编辑器配置 | 不需要操作 | 完全忽略 |
| `.github/workflows/` | CI/CD 配置 | 不需要操作 | 完全忽略 |
| `.gitignore` | Git 忽略规则 | 不需要操作 | 完全忽略 |
| `package.json` / `requirements.txt` | 依赖声明 | 知道项目用了哪些依赖 | 不需要学包管理器语法 |

**铁律**：遇到上面这些文件的语法看不懂时，**不是你的问题，是那一层的知识你还没学**。跳过，继续读核心代码。

## 常见误区

> [!warning] "我要先学 CMake/Makefile 再开始读代码"

不需要。构建工具是**工具知识**，编程是**领域知识**。工具知识应该在实际需要时边用边学，不需要提前系统学习。你现在需要的只是 `mkdir build && cd build && cmake .. && make` 这个咒语。理解它在做什么可以等到第三个项目之后。

> [!warning] "这个项目用了 15 个 .h 文件我看不懂"

`.h` 文件是头文件——声明接口用的。读代码时从 `.cpp` 文件（实现）开始，`.h` 文件只在你用到一个函数想知道它的参数类型时才看。
