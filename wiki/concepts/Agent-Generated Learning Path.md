# Agent-Generated Learning Path

> [!summary] 自写摘要
> 当 GitHub 仓库没有配套教程时，Agent 可以从代码依赖图和学习路径，替代作者教程的角色。核心变化：验证责任从作者转移到学习者——Agent 的理解是待验证的假设，学习者通过动手（改代码、写测试）来确认。四步工作流：代码侦查 → 路径生成 → 交互学习 → 交叉验证。关键原则：先自解释再看 Agent 解释；卸载存储可以，卸载验证不行。

---

## 问题起源

educ8s 的 [[cpp-tetris-raylib|C++ Tetris 教程]] 有配套视频逐行讲解——但大多数 GitHub 仓库没有。当教程用完时（学完 Tetris 之后），Agent 能否替代作者教程的角色，为任意 GitHub 仓库生成代码学习路径？

**答案：可行，但验证责任发生了根本转移。**

---

## 作者教程 vs Agent 路径：能力对比

| 维度 | 作者教程 | Agent 生成路径 |
|------|---------|---------------|
| **代码事实** | 100% 准确（作者写的） | 需验证（Agent 可能读错） |
| **设计意图** | 知道"为什么这样做" | 只能推理，标记为"推测" |
| **依赖排序** | 知道真实构建顺序 | 从 `#include` 和调用图反推 |
| **复杂度梯度** | 作者刻意设计 | 从函数行数和依赖深度估算 |
| **踩坑预警** | 作者知道常见错误 | 只能猜，除非有 issue 讨论 |
| **教学节奏** | 人类经验 | 偏机械化，可能跳跃太大 |

**核心差异**：作者教程 = **设计意图 + 代码事实**。Agent 路径 = **代码事实 + 推理**。

## 四步工作流

```
┌─ 有教程 ───────────────────────────┐
│ 作者定顺序 → 跟着看 → 写代码验证    │
└────────────────────────────────────┘

┌─ 无教程（Agent 生成）──────────────┐
│ Agent 侦查 → Agent 生成路径         │
│   → 你按路径读 + 自解释              │
│   → 你动手验证 Agent 的理解对不对    │
└────────────────────────────────────┘
```

### Step 1: 代码侦查

Agent 一次性读完全部源文件，输出三样东西：

```
1. 文件依赖图（谁 include 了谁，谁调用了谁）
2. 概念清单（有哪些类/函数/数据结构）
3. 复杂度估计（函数行数、嵌套深度、外部依赖）
```

纯机械工作，Agent 准确率很高。不做教学判断，只做事实提取。

### Step 2: 生成学习路径

基于依赖图，按以下规则排序：

| 排序规则 | 示例 |
|---------|------|
| 被依赖 → 先学 | Grid 被 Game 依赖 → Grid 在前 |
| 行数少 → 先学 | 50 行函数 → 200 行函数 |
| 无外部依赖 → 先学 | 纯数据结构 → 调用 raylib API 的 |
| 有视觉反馈 → 穿插 | 读完一个模块就跑一下看效果 |

对 Tetris 项目，Agent 生成的路径会**非常接近** educ8s 的 12 步——因为这是代码依赖关系决定的，不是作者随机选的。

### Step 3: 交互学习

每步的学习循环：

```
1. Agent: "今天读 Grid.cpp，重点关注这 3 个函数"
2. 你: 读代码 + 写自解释（用中文讲这段在做什么）
3. Agent: 读你的自解释，指出理解偏差
4. 你: 改一行代码，跑一下，看行为变化
5. Agent: "你改的是 X，为什么 Y 变了？"（苏格拉底式追问）
6. 重复，直到这个文件你讲得通
```

与 [[Code Reading as Learning Strategy]] 的六步法一致——只是多了 Agent 做解释器和追问者。

### Step 4: 交叉验证（最关键）

**必须在每步结束时动手验证**：

| 验证方式 | 验证了什么 |
|---------|-----------|
| 改一个参数，看行为变化 | Agent 对这个参数作用的理解对不对 |
| 注释掉一行，看哪里崩 | Agent 对这行代码依赖关系的理解对不对 |
| 给一个函数写测试 | **你**对这个函数的理解对不对 |
| 问 Agent "如果我想让方块加速下落，改哪里" | Agent 是否真的理解代码结构 |

> [!key-insight] 根本转移
> 有教程时你信任作者的理解。没有教程时，Agent 的理解是一个**待验证的假设**——你通过动手（改了能跑、测试通过）来确认 Agent 没把你带沟里。

## 关键原则

### 1. 先自解释，再看 Agent 解释

顺序不能反。如果先看 Agent 的解释，你会产生"我懂了"的错觉（流利性效应）。先自己挣扎，再对照 Agent——这才是 [[合意困难-Desirable-Difficulties|含意困难]] 的正确应用。

### 2. 卸载存储，不卸载验证

对应 [[认知卸载-Cognitive-Offloading]] 的核心区分：
- ✅ 卸载给 Agent：代码依赖图、概念清单、学习顺序提议
- ❌ 不能卸载：判断 Agent 的理解对不对、确认自己真的懂了

> Agent 的理解 + 你动手验证 = 真知识。Agent 的理解 + 你只看不碰 = 幻觉传递。

### 3. "设计意图"一律标推测

Agent 说的"作者这样设计是因为 X"——除非有注释或 commit message 佐证，一律在前面加"推测:"。这在笔记中标记为：

```markdown
> [!note] 推测
> Agent 推测：旋转用状态循环而非矩阵是因为作者想避免数学门槛。
> 未在源码注释中确认。
```

## 风险与对策

| 风险 | 可能性 | 对策 |
|------|--------|------|
| Agent 读错代码逻辑 | 中 | 每步必须跑代码验证 |
| Agent 低估/高估难度 | 中 | 卡住超过 30 分钟 → 跳过，标记"待回访" |
| 路径顺序不合理 | 低 | 依赖图是机械的——你可以质疑排序依据 |
| Agent 编造设计意图 | 中 | 一律标记为"推测" |
| 过度依赖 Agent，不自解释 | **高** | 先自解释，再看 Agent 解释——顺序不能反 |

## 从"有教程"到"无教程"的过渡

educ8s 的 Tetris 教程是**脚手架**，不是**轮椅**。过渡路径：

```
educ8s Pong（有教程）
  → educ8s Snake（有教程，你已经完成）
    → educ8s Tetris（有教程，当前）
      → 下一个 GitHub 仓库（无教程）
        → 启动 Agent 生成路径模式 ← 你在这里切换
```

学完 Tetris 之后，你应该**刻意**选一个没有教程的仓库，用 Agent 生成路径——这是从"跟随教程"到"独立学习"的跳跃。
